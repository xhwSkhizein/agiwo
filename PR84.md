# PR #84 总览

## 一句话说明

这个 PR 做的核心事情很简单：

**把旧的“retrospect（事后回顾）”机制，替换成了一个更主动的“Goal-Directed Review + Step-Back（目标导向回顾 + 退一步整理上下文）”机制。**

以前系统更多是在“内容太多了”“轮数够多了”之后，才想办法回顾一下。
这次改完以后，Agent 会先声明目标拆解，系统再在关键节点强制它检查“我现在做的事情，到底是不是还在推进当前目标”。

如果发现方向偏了，系统不会粗暴地重建上下文，而是把偏题那段冗长的 tool 输出，整理成一段简短的“经验总结”，这样既保留了做过什么，也减少了噪音。

---

## 这次主要改了什么

可以把这次改动理解成 5 件事。

### 1. 新增了一套 review 子系统

新增了 `agiwo/agent/review/` 包，里面把这套机制拆成了几个很清楚的角色：

- `goal_manager.py`
  负责管理 milestone，也就是“阶段性子目标”。
- `review_enforcer.py`
  负责判断什么时候应该触发 review。
- `step_back_executor.py`
  负责在确认“跑偏”之后，做一次安全的上下文整理。
- `__init__.py`
  里面的 `ReviewBatch` 把前面几块串起来，供 `run_tool_batch.py` 在运行时直接使用。

换句话说，这次不是在旧逻辑上打补丁，而是把“目标管理、触发时机、回顾执行、上下文整理”正式做成了一条完整链路。

### 2. 用 milestone 取代“模糊地做任务”

现在 Agent 被要求先调用 `declare_milestones`，把一个大任务拆成几个更具体、更可验证的小目标。

比如不是说“理解代码”，而是说：

- 找出 token 是怎么校验的
- 定位 bug 根因
- 修复并验证

这样系统后面做 review 时，就不再是问一个很空泛的问题，而是问：

**“你最近做的这些步骤，到底有没有推进当前这个 milestone？”**

这会让 Agent 的工作方向更可控，也更容易判断“现在是不是已经偏题了”。

### 3. 用强制 review 取代“想起来才回顾”

这次新增了 `review_trajectory` 工具。

系统会在一些关键时机，往 tool result 里注入一个 `<system-review>` 提示，要求 Agent 必须做一次自检。

触发 review 的时机主要有三类：

- 做到一定步数后
- 连续出错后
- milestone 切换后

Agent 这时要明确回答两件事之一：

- `aligned=true`
  说明现在方向对，继续往前。
- `aligned=false`
  说明刚才这段探索偏了，并给出一段简洁的经验总结。

这里的关键点是：**review 不再依赖 Agent 自觉，而是由系统强制触发。**

### 4. 用 step-back 整理“偏题痕迹”，而不是简单删除历史

这部分是这次最重要的工程设计点之一。

当 Agent 说 `aligned=false` 时，系统会执行一次 `step-back`。

它做的事情不是“把历史删掉重来”，而是：

- 保留原来的 tool call
- 保留整体消息顺序
- 只把 checkpoint 之后那些偏题的 tool result 内容，替换成一段经验总结

也就是说，系统保留“你做过什么”的轨迹，但把“那次调用返回的几千 token 冗长输出”压成一句更有价值的话。

这有两个直接好处：

第一，后面的模型还能知道“这条路已经试过了，不用再重复踩坑”。

第二，上下文不会继续被大段低价值输出占满。

### 5. 把这套机制接入了运行时、日志、流式事件和 Console

这次改动不只是 Agent 内部行为变化，还把这个变化变成了“可观察、可回放、可展示”的系统事实。

主要包括：

- 新增 `StepBackApplied` 这类 run-log fact
- 更新 stream event，让外部能看到 step-back 发生过
- 更新 runtime decision view
- Console 会把这些决策事件展示出来

所以这次不是单纯“prompt 变了”或者“工具变了”，而是从执行逻辑到观测面都跟着一起升级了。

## 这次改动是怎么工作的

如果把整个过程用最朴素的话讲，大概是这样：

### 第一步：先把任务拆清楚

Agent 开始工作时，先用 `declare_milestones` 说清楚：

- 当前任务要分哪几个阶段做
- 每个阶段做到什么才算完成

系统会把这些 milestone 存进 `RunLedger.review` 里，后续 review 都基于这个状态来判断。

### 第二步：正常执行工具

Agent 照常调用搜索、读文件、执行命令之类的工具。

这部分整体执行入口仍然是 `run_tool_batch.py`，只是现在每一批工具结果都会先经过 `ReviewBatch.process_result()`。

### 第三步：系统判断要不要强制 review

如果最近做了太多步、连续出错，或者里程碑切换了，系统就会在当前 tool result 后面附上一段 `<system-review>` 提示。

这相当于系统在说：

**“停一下，先别继续冲，先回答你现在是不是还在推进当前目标。”**

### 第四步：Agent 给出对齐判断

这时 Agent 必须调用 `review_trajectory`。

如果它说 `aligned=true`：

- 系统记录一个 checkpoint
- 清掉这次临时 review 元数据
- 继续往下走

如果它说 `aligned=false`：

- 系统记录这段经验总结
- 触发 `step_back`
- 把刚才跑偏阶段的大段 tool 输出压缩成这段经验

### 第五步：后续上下文变得更干净

最终效果不是“失忆”，而是“保留轨迹、压缩噪音”。

模型后面还能知道：

- 之前试过哪条路
- 那条路为什么不行
- 下一步应该往哪收敛

这比“直接删掉消息”更稳，也比“什么都留着”更节省上下文。

## 对 Agent 来说，最大的好处是什么

### 1. 更不容易跑偏

以前 Agent 可能会在一个不太相关的方向上越走越远，直到上下文爆掉，或者人类回头看日志才发现它偏了。

现在系统会定期把它拉回来问一次：

**“你现在做的事，真的在推进当前目标吗？”**

这会明显减少那种“忙了很多，但离目标没更近”的情况。

### 2. 更容易从失败里留下真正有用的东西

旧模式里，一次失败探索通常只会留下很多低价值输出。

新模式里，失败探索会被提炼成一句经验：

- 试了什么
- 学到了什么
- 下一步应该怎么调整

这对 Agent 很重要，因为模型真正需要的不是“原始噪音”，而是“下一步决策所需的信息”。

### 3. 上下文质量更高

这次不是只做压缩，而是在做“语义筛选”。

保留下来的内容更接近：

- 当前目标是什么
- 哪条路已经试过
- 哪条路值得继续

对长任务来说，这比单纯按 token 数做截断或压缩更有价值。

### 4. 保留意图链条，减少副作用

这次 step-back 的实现刻意遵守一个很重要的原则：

**尽量不破坏已有 message 结构。**

它不随便删除业务消息，也不重排历史，而是只替换 tool result 的内容。

这样做的价值是：

- 对模型上下文更稳定
- 对 KV cache 更友好
- 对调试更友好
- 对 replay / observability 更一致

### 5. Agent 的“工作方式”更像一个会阶段性校准的人

这次改动让 Agent 的行为从：

“一路往前做，做多了再回头看看”

更接近：

“先拆目标，做一段就校准一次，偏了就总结经验再继续”

这是一个很明显的行为升级。

## 这次改动对系统的价值是什么

除了让 Agent 更聚焦，这个 PR 对整个系统还有几层更底层的价值。

### 1. 从“基于大小的回顾”升级成“基于目标的回顾”

旧 retrospect 更像是在回答：

**“内容是不是太大了？”**

新机制回答的是：

**“这些内容是不是还在推进目标？”**

这两者看起来相似，其实判断标准完全不一样。

前者更偏资源管理，后者更偏任务质量控制。

### 2. 把 review 变成显式能力，而不是隐式习惯

这次通过 `declare_milestones` 和 `review_trajectory` 两个工具，把这件事做成了明确协议。

好处是：

- prompt 更清楚
- 运行时更好接入
- 测试更容易写
- 行为更可验证

### 3. 可观测性更强

现在 step-back 不只是“内部发生过”，而是会进入：

- run log
- stream event
- runtime decision view
- Console 的 session observability

这意味着后续无论是调试、排障、做产品展示，还是做数据分析，都会比以前更清楚。

### 4. 为长任务、多阶段任务打基础

这次加 milestone，本质上是在给 Agent 引入“阶段管理”。

这对简单任务可能感受不强，但对下面这些场景会很重要：

- 多步骤修 bug
- 大范围代码探索
- scheduler 驱动的长任务
- 需要在多个子目标间切换的任务

它让系统不再只知道“任务开始了”，而开始知道“任务现在推进到哪个阶段了”。

## 这个 PR 会怎样影响现有行为

这个 PR 不是只加了几个新文件，它会真实改变 Agent 的工作流。

### 对 Agent 行为的影响

- Agent 会被 prompt 明确要求先声明 milestone
- 系统会在关键节点强制要求 Agent 做 review
- 当 Agent 承认偏题后，系统会自动整理刚才那段上下文

也就是说，Agent 的运行从“连续执行”变成了“连续执行 + 阶段性校准”。

### 对运行时的影响

- `run_tool_batch.py` 现在会在每批 tool results 后跑 review 逻辑
- `RunLedger` 上新增了 `ReviewState`
- step-back 会写入 canonical run-log fact
- replay、serialization、trace、stream 都跟着对齐

### 对 Console / 可观测性的影响

- Console 看到的不再只有 compaction / termination
- 现在还能看到 step-back 这种运行时决策
- 这会让 session detail 更接近“真实发生了什么”

### 对配置和接口的影响

- 旧的 retrospect 配置被替换成新的 review 配置
- 新增 `enable_goal_directed_review`
- `review_step_interval` 和 `review_on_error` 成为新的控制项

这意味着系统的语义也变了：
不是“要不要做 retrospect”，而是“要不要启用目标导向 review”。

---

## 如果让我用最朴素的话总结这次 PR

这次 PR 做的不是“让 Agent 更会压缩上下文”，而是：

**让 Agent 在执行过程中，越来越像一个会先拆目标、途中会校准、跑偏了会总结经验再继续的人。**

它带来的直接价值是：

- 减少偏题
- 提高上下文质量
- 保留有价值的失败经验
- 让运行时决策更可观察
- 为长任务和多阶段任务建立更稳的工作方式

如果只记住一句话，可以记这个：

**以前系统主要关心“内容是不是太多了”，现在系统开始关心“这些内容是不是还在推进目标”。**
