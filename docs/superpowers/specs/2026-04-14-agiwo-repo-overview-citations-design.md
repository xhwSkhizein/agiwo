# Agiwo Repository Overview Citations Design

## Summary

The current public repository overview page explains Agiwo at the directory and architecture level, but its evidence model is still weak. It provides a small global `Source Pointers` list, yet individual statements about modules, runtime surfaces, and boundaries are not directly tied to their underlying sources.

This design adds source-aware citations to the public repository overview so each important claim is easier to verify. The page should move closer to a code-aware repository explainer: readers can see what a module does, why that statement is there, and which repository paths or docs support it.

## Goals

- Attach compact source pointers directly to the main repository overview entities.
- Keep citations deterministic and rule-driven rather than generated by an LLM.
- Improve reader trust by making each architecture claim easier to verify.
- Preserve page readability by combining inline citations with section-level source rollups.
- Strengthen SEO by exposing more stable repository terminology, path names, and supporting document references in the rendered HTML.

## Non-Goals

- Line-level or permalink-style citations.
- Function-level or symbol-level source attribution.
- Git history, blame, or commit-based provenance.
- Automatic regeneration on every push.
- A generalized citation system for every docs page in the public site.

## Constraints And Assumptions

- The public docs site remains static and separate from the internal `console/web` app.
- Citation data must be generated ahead of time into committed structured data.
- The generator should only use stable repository facts and selected docs already committed in the repo.
- The first version should prefer low-noise, high-confidence citations over deep attribution coverage.
- Existing repo overview content is already code-tree-driven and should remain that way.

## Recommended Approach

Use a hybrid citation model:

1. Add short inline citation chips to each important overview item.
2. Add a deduplicated `Sources` block at the end of each section.
3. Keep a final reference-documents section for broader navigation.

Why this approach:

- Inline citations keep evidence attached to the statement being read.
- Section-level rollups reduce the need to repeat the same source pointer many times.
- The final reference section still gives visitors a clean list of key repository documents.

Rejected alternatives:

- Section-only sources: clean layout, but weak evidence-to-claim binding.
- Inline-only citations: strong attribution, but too repetitive and visually noisy.

## Citation Model

Each rendered entity on `/docs/repo-overview/` should carry explicit citation metadata.

At minimum, the generated data should support citations for:

- layout items
- runtime surfaces
- architectural boundaries

Each citation entry should contain stable, display-ready metadata such as:

- `label`
- `path`
- optional `title`
- `kind`

Recommended `kind` values:

- `directory`
- `entrypoint`
- `document`

The generator output should remain simple JSON and should not require any build-time repository scanning from Astro.

## Source Types

### Code-Tree Sources

Use directory paths when the statement is fundamentally about repository structure or ownership.

Examples:

- `agiwo/agent`
- `agiwo/scheduler`
- `console/server/services`

### Entrypoint Sources

Use public entry files when the statement is about runtime surface or public API orientation.

Examples:

- `agiwo/agent/__init__.py`
- `agiwo/agent/types.py`
- `agiwo/llm/factory.py`

### Document Sources

Use repository docs when the statement reflects stable architecture or onboarding guidance rather than a single code directory.

Examples:

- `AGENTS.md`
- `README.md`
- `docs/architecture/overview.md`
- `docs/getting-started.md`
- `docs/guides/multi-agent.md`

## Source Priority

Citation selection should prefer the most direct and stable evidence:

1. concrete repository path for the subject being described
2. public entrypoint file when the subject is a runtime surface
3. `AGENTS.md` for stable directory responsibilities and architecture boundaries
4. `README.md` and public docs for broader explanatory context

The generator should not attach a citation if it cannot do so with a clear rule.

## Rendering Strategy

### Inline Citations

Each overview item should render a short set of citation chips immediately after its description.

Examples of where inline citations should appear:

- each directory responsibility in layout sections
- each runtime surface entry
- each architecture boundary statement

Inline citations should be compact. They are evidence markers, not full annotations.

### Section Sources

Each major section should render a deduplicated `Sources` list built from the citations used inside that section.

This gives readers a second, cleaner way to inspect the supporting material without scanning each inline chip.

### Global Reference Section

The current `Source Pointers` section should be repositioned as a broader `Reference Documents` section.

Its role should be:

- key repository reading list
- related architecture docs
- onboarding links for visitors who want more context

It should no longer be the only source-citation mechanism on the page.

## Generator Changes

Update `scripts/generate_repo_overview.py` so the generated payload includes citation arrays attached to each relevant entity.

Expected changes:

- layout items gain `citations`
- runtime surfaces gain `citations`
- boundaries become structured objects with `text` and `citations`
- the existing top-level reference list is retained for document navigation

The generator should use stable mapping rules instead of heuristic text analysis.

Examples:

- a layout item for `agiwo/agent` should cite `agiwo/agent` and optionally `AGENTS.md`
- a runtime surface for `agiwo.agent` should cite `agiwo/agent`, `agiwo/agent/__init__.py`, and optionally `agiwo/agent/types.py`
- a boundary statement about scheduler direction should cite `agiwo/scheduler` and `AGENTS.md`

## UI Changes

The rendering layer should add small reusable citation UI primitives rather than embedding markup ad hoc inside every repo overview component.

Recommended rendering structure:

- one citation-chip component for inline display
- one section-sources component for deduplicated rollups
- small updates to existing repo overview section components to accept citation-bearing data

This should preserve the existing page structure while making source attribution obvious.

## SEO Impact

This work should strengthen the public page in three ways:

- more stable repository paths appear in visible HTML near the claims they support
- more architecture terminology is reinforced by repeated, structured references
- the overview becomes more trustworthy and useful as a landing page for branded and non-branded technical queries

The goal is not citation markup for search engines. The goal is better human-verifiable technical content, which should indirectly improve search quality signals.

## Risks

- Too many inline citations can make the page feel cluttered.
- Over-citing `AGENTS.md` can make the page look document-driven instead of code-aware.
- Weak citation rules can create misleading evidence if a source is only loosely related to a claim.
- If the citation UI is inconsistent, the page may feel heavier without becoming more useful.

## Success Criteria

- Every layout item, runtime surface, and boundary note on `/docs/repo-overview/` has direct source-aware citation support.
- Each major section exposes a deduplicated `Sources` rollup.
- The page keeps the existing architecture-level readability and does not collapse into a footnote wall.
- Citation generation remains deterministic and manually refreshable.
- Visitors can verify major claims without leaving the page guessing where the explanation came from.
