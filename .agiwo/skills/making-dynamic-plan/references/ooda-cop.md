# OODA-COP 动态规划元方法论

核心隐喻：计划不是静态文本，而是**能感知环境、自我突变、淘汰劣势、强化优势的生命体**。

四个层层嵌套的模块，适应任何背景、任何人群、任何问题。

---

## 模块一：认知引擎 — 感知与重构

**适配变量：背景（未知程度 / 风险等级）**

### OODA 循环升级版

基于博伊德的 OODA 逻辑，核心在于 **观察-定向-决策-行动** 的循环速度。

- **观察**：区分"已知的已知、已知的未知、未知的未知"
- **定向**（核心）：选择观察框架 — 把问题看作"机械故障"还是"生态系统失衡"？定向错了，后面全错
- **适应性策略**：
  - 高确定性（如建筑工程）→ **计划驱动**，重前期设计
  - 中不确定性（如产品开发）→ **价值驱动**，重优先级排序
  - 高不确定性（如创业、科研）→ **假设驱动**，计划即实验清单

**交付物**：《情境诊断书》— 当前问题的复杂度层级 + 风险对冲策略

## 模块二：结构引擎 — 拆解与编织

**适配变量：问题（类型 / 规模 / 依赖关系）**

### 三级结构化拆解

**第一级：目标澄清 (PSA)**
- P（问题）：解决谁的什么痛点？
- S（成功标准）：哪三个量化指标衡量完成？
- A（边界）：哪些事坚决不做？

**第二级：系统建模**
- 线性问题（如生产流程）→ PERT 网络图 / 关键路径法
- 复杂问题（如组织变革）→ Cynefin 框架，先探测再规模化
- 混沌问题（如危机公关）→ 先稳定再分析，顺序不能反

**第三级：颗粒度裁剪**

引入"管理摩擦力"概念 — 计划越细，维护成本越高。

- 专业人士 → 只需目标 + 约束条件
- 新手 / 跨部门 → 拆解到"下一步行动"级（15 分钟内可完成）
- 公式：`计划颗粒度 ∝ 执行者经验值 × 任务耦合度`

**交付物**：《动态工作分解结构》— 有逻辑依赖关系的任务图谱

## 模块三：进化引擎 — 反馈与自修正

**适配变量：时间流逝（变化速率 / 信息更新）**

### 双回路学习与节奏设计

**反馈回路双循环：**
- 单回路：是否按计划走？→ 修正行动偏差
- 双回路：计划的前提假设还成立吗？→ 修正目标偏差
- 机制：预设"检验点" — 不是检查进度，而是检查"当初做这件事的理由是否还成立"

**动态节奏调频：**
- 探索期 → 短周期迭代（按周或双周），快速试错
- 执行期 → 中周期跟踪（按月），保持稳定
- 收尾期 → 长周期复盘，沉淀资产
- 关键原则：**反馈频率必须高于环境变化频率**

**缓冲与冗余的动态分配：**

借鉴关键链管理 — 不把安全时间藏在每个任务里（帕金森定律），而是集中建立缓冲池。根据任务完成概率动态分配项目缓冲、接驳缓冲。缓冲消耗过半时触发预警。

**交付物**：带有传感器的计划系统 — 自动识别需要"调整动作"还是"更换地图"

## 模块四：适配引擎 — 人性与意义

**适配变量：人（认知风格 / 动机状态 / 关系网络）**

### 认知风格匹配与意义建构

**计划表达形式的适配：**
- 分析型（工程师、财务）→ 数据、逻辑、因果链
- 概念型（创意、市场）→ 愿景、故事、可能性
- 结构型（行政、运营）→ 清单、流程图、检查表
- 社交型（销售、HR）→ 角色、互动、情感价值

同一套计划内核，可以穿不同的外衣。

**控制感的赋予：**

人对"自己参与制定的计划"忠诚度远高于"被分发的任务"。即使大方向已定，也应在路径选择、节奏安排、资源调配中留出自主空间。最低限度：让执行者参与估算时间、确定里程碑。

**意义感的注入：**

将任务语言转化为角色语言。不是"你需要写代码"，而是"你作为架构师，正在构建支撑百万用户的基础设施"。让执行者看见自己的产出如何帮助了具体的某个人。

**交付物**：人格化的计划书 + 持续的共识对齐机制

---

## 四步启动流

1. **定调**：用 1 小时完成情境诊断，确定复杂度等级，选择主导策略
2. **塑形**：用 PSA 框定边界，用 WBS 拆解至适配颗粒度，识别关键路径与缓冲区域
3. **布点**：在时间轴上设置双回路检验点 — 不仅问"进度如何"，更问"假设是否依然成立"
4. **适配**：将计划翻译成接收者能共振的语言，留出参与空间，绑定角色身份
